# Phase 04 — UI Design Contract

> Visual and interaction contract for Phase 04: Content Expansion.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> This phase adds 9 pages (×4 locales): 3 retrofit comparison pages + 3 category-alternative
> comparison pages + 3 guide pages. No net-new UI primitives. All components reuse the
> established pattern library. This contract codifies the exact visual rules for executor fidelity.

## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, separator dots |
| sm | 8px | Compact element spacing, tag/label padding |
| md | 16px | Default element spacing, table cell padding (`px-3 py-3` = 12/12 → round to md) |
| lg | 24px | Section padding, card internal spacing (`py-12` on related-service section) |
| xl | 32px | Layout gaps between major elements |
| 2xl | 48px | Vertical section padding on desktop (`py-16 md:py-24` base rhythm) |
| 3xl | 64px | Page-level spacing, hero areas |

**Container constraint:** `mx-auto max-w-3xl px-6` (content) and `mx-auto max-w-4xl px-6` (comparison table section). Use `max-w-3xl` for prose sections (verdict, guide body), `max-w-4xl` for table sections. **Do not exceed `max-w-4xl`.**

**Exceptions:**
- Touch targets (StickyCtaBar buttons): `min-h-[44px]` — mandatory 44px minimum per WCAG 2.5.5. Source: `StickyCtaBar.tsx`.
- Section alternation: fog-band sections use `py-16 md:py-24` (top/bottom) consistently; white sections use `py-12` for linking/CTA sections. Match existing comparison page rhythm exactly.

## Interaction Contract

### Reveal / Progressive Disclosure — CRITICAL INVARIANT

**The answer-first verdict/answer block MUST NOT be wrapped in `<Reveal>` or any component that sets `opacity:0` on first paint.**

Source: Phase 2 D-03 (crawler-visibility invariant), CONTEXT.md D-11, P2 CONTEXT.md invariant "no content behind Reveal/opacity:0".

Rule: The first `<p>` (verdict for comparisons, answer block for guides) renders as bare SSR HTML with no animation wrapper. `<Reveal>` is permitted on all secondary sections (table, FAQ, CTA, related service link, back nav) but NEVER on the opening answer block.

### `<Reveal>` usage pattern (non-answer sections)

Match existing comparison page rhythm exactly:
- `<Reveal>` wraps nav link, h1, intro paragraphs section, table, verdict h2+p, related-service section, each FAQ item, CTA section.
- `delay` increments: 0 → 0.05 → 0.1 (for staggered sequences). Do not exceed 0.1 delay on any single page.
- Do NOT add Reveal to TrustBand (hardcoded rule in `TrustBand.tsx`).

### KeyPageChrome mounting

Mount `<KeyPageChrome>` on every new comparison and guide page:
- Pass `showTrustBand={true}` (default) on all new pages — they are not the home page.
- Mount ONCE per page. Never in layout. Guard against double-mount per `6a4295e` fix.
- `consentKnown` propagated from `readConsent()` in the page's server component (same as other key pages).
- StickyCtaBar suppresses itself when `consentKnown=false` (first visit, consent bar visible).

### RTL support

- `dirFor(lang)` returns `"rtl"` for `ar`.
- Back nav arrow reverses: `{dirFor(lang) === "rtl" ? "→" : "←"}`.
- StickyCtaBar uses `flex-row-reverse` for RTL.
- ComparisonTable: `text-left` on the table; no RTL override needed (CSS logical properties handle it at browser level via `dir` on `<html>`).
- No new RTL CSS is required — existing pattern covers all cases.

### Table accessibility

- `ComparisonTable` renders a full `<table>` with `<caption className="sr-only">`, `scope="col"` on `<th>` in thead, `scope="row"` on first `<th>` in tbody rows.
- No changes to this pattern. Extend by passing the same `columns` / `rows` / `caption` props.

### Internal links

- Every new page links forward to its related service money-page (`servicePath(service, lang)`).
- Every service page lists related comparisons/guides via `comparisonsForService()` + the new `guidesForService()` mirror.
- Use `<Link>` (Next.js) for all internal navigation. Use `<a href="tel:">` for phone only.

## Schema / Structured Data Contract

This section governs the visual output of JSON-LD (invisible to users but part of the page contract).

| Page type | Schema emitted | Builder location |
|-----------|---------------|-----------------|
| Comparison (retrofit + category-alternatives) | Product + Review (`aggregateRating` gated on `reviewsFetchedAt`) + Breadcrumb | `seo.ts` — add `productGraph` + `reviewGraph`; `breadcrumbGraph` exists |
| Guide pages | Article + Breadcrumb | `seo.ts` — add `articleGraph`; `breadcrumbGraph` exists |

**Live-data gate:** `Review` / `aggregateRating` emitted only when `reviewsFetchedAt` is truthy (SCHEMA-03 pattern from `site.reviews`). Never emit a self-authored or placeholder rating.

**No HowTo schema.** Locked: roadmap success criterion 2 and REQUIREMENTS.md framing. Do not add.

**No FAQPage schema** on comparison pages going forward. The existing `faqPageGraph` call in the current comparison page will be replaced by Product + Review + Breadcrumb per D-04. Remove `faqPageGraph` from comparison pages during retrofit.

## Crawler-Visibility Invariant (Phase 2 carry-forward)

The following rules are absolute and apply to every page touched in this phase:

1. **Answer-first content in SSR HTML.** `curl https://onglessanssouci.com/en/<slug> | grep -i "<verdict-phrase>"` must return a match. Verify before merge.
2. **No `opacity:0` on content at first paint.** `<Reveal>` is permitted on decorative/secondary sections only.
3. **No content exclusive to client JS.** All factual copy (verdict, answer block, table, FAQ) must render in the SSR `<body>` before hydration.
4. **`[PRICE:*]` tokens must not reach production.** A build-fail gate (vitest assertion or CI grep on `.json` dict files) blocks merge if any `[PRICE:*]` substring is present in the built dictionary output.
